# HTST prefactor profiling — results

Generated by `run_profile.py` + `compare.py` on the local macOS dev machine
(LAMMPS 20250722 with PHONON; custom OpenMPI), 5 repeats/point, `dx=fd_step=0.01`.
Reproduce: see `README.md`. Per-Hessian CSVs are written to `results/`
(gitignored machine-specific output; regenerate from the README commands).

**Verdict: use `eskm` (LAMMPS `dynamical_matrix`).** It is faster in serial wall
time at every radius AND needs ~2 orders of magnitude fewer engine round-trips —
the latter is what dominates once the prefactor runs through a remote/global
engine (the Option A backend). The two modes agree on ν₀ to 0.014% wherever the
frozen-boundary cluster is physical.

ν₀ is `—` (None) at r≥6 because the 86/105-atom clusters **saturate**: the free
region pulls in the finite cluster's under-coordinated boundary atoms, the
frozen-boundary assumption breaks, and the saddle no longer has a clean single
negative mode. The **timing** comparison stays valid (both modes do the same
work); the physical regime for these clusters is r≲4.

## Recommendation for the Option A backend

- **eskm @ r=6.0**: serial-fastest by 3.0× (ni100) / 4.6× (ni4000).
- **Round-trips/event: eskm = 3, fd = 18·n_free** (882 / 1404 at r=6). Through the
  global engine each round-trip is a messenger send/recv, so eskm wins by ~300–470×
  on traffic alone — independent of local compute. This is the decisive factor:
  the Option A `HtstBackend.compute()` should call
  `manager.global_compute_dynamical_matrix(...)` (the eskm path), 3 round-trips per
  event (min1/saddle/min2), not an FD force loop.
- eskm's cost is dominated by the in-LAMMPS `dynamical_matrix` C++ FD; the file
  read is negligible (see the `t_dynmat_cmd` vs `t_file_read` columns in the CSVs).
